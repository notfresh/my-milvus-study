# Atlas — 什么是 Atlas，为什么要有 Atlas

## 起源

Atlas 诞生于一个简单的观察：大型项目的知识是**散落**的——架构藏在代码里，设计藏在 design doc 里，决策的「为什么」藏在 git commit message 里，而新来的 agent（或者三个月后的自己）打开项目时，什么都不知道。

`docs/` 里有设计文档，但那是正式的、面向所有人的。`CLAUDE.md` 里有规则，但那是静态的、不能解释「这个子系统为什么长这样」。这两者之间，有一个巨大的空白地带：**那些「读懂了、踩过坑、验证过」之后才会知道的东西**。

Atlas 就是填这个空白的。

## Atlas 是什么

Atlas 是**项目的认知地图**。

它的核心职责是回答两类问题：

1. **「这是什么？」** — 子系统的浓缩视图、工作流程的关键步骤、文件结构的解读。
2. **「为什么这样？」** — 做某个技术选型时的权衡、在多个方案中选了 A 而不是 B 的理由。

它不是：

- `docs/` 的替代品。Atlas 里的内容比 docs 更轻、更个人、更快速迭代。
  docs 是「写给所有人看的正式文档」，Atlas 是「带协作痕迹的速记本」。
- `CLAUDE.md` 的补充。CLAUDE.md 告诉你「做什么、怎么做（规则层面）」，
  Atlas 告诉你「为什么这样做（认知层面）」。
- 代码的替代品。Atlas 里的描述永远要能追溯到源码；描述和代码对不上的地方，
  以代码为准，先改 Atlas。

## 核心设计原则

### 1. 单入口

`Atlas/INDEX.md` 是唯一入口。任何人 / agent 想了解项目的某个面，
先翻这里。INDEX.md 列出所有知识条目，任何新增内容都必须同步更新 INDEX.md。

### 2. 两种文件类型，两种命名

| 类型 | 命名格式 | 例子 | 特征 |
|------|----------|------|------|
| 持续维护的知识 | `topic-subtopic.md` | `dependency-landscape.md` | 条目随项目演进不断追加 / 修正 |
| 一次性事件 / 决策 | `YYYY-MM-DD-title.md` | `2026-06-12-conan-vs-vcpkg.md` | 写完即定稿，不再改动 |

不要混用。如果一个知识条目变成了「记录某次决策」，改名为日期格式。

### 3. Frontmatter 是必需的

每个文件开头必须有这三行：

```yaml
---
topic: build | subsystem | convention | decision | meta
last-verified: YYYY-MM-DD
status: draft | stable | stale
---
```

- **`topic`**：分类。统一分类让 INDEX.md 可以按 topic 分组显示。
- **`last-verified`**：上次核实日期。超过 3 个月的条目，引用前应重看。
  AI agent 的上下文是有上限的，看到旧日期就知道需要重新读取。
- **`status`**：draft → stable → stale。stable 是目标状态，
  代表「已交叉验证，可以信任」。

### 4. 追加优先，新建次之

已有条目的内容应该**追加**到对应文件，而不是建新文件。
INDEX.md 是唯一需要同步更新的地方。

> 好的做法：`dependency-landscape.md` 里加一节「如何新增 C++ 依赖」
> 坏的做法：新建 `how-to-add-cpp-dep.md`，然后 INDEX.md 里多一条孤零零的链接

### 5. 与源资源的链接永远要保留

每篇 Atlas 条目里涉及到具体文件时，用相对路径标注（如 `../internal/core/conanfile.py`）。
每篇 Atlas 条目的 `related` frontmatter 字段列出相关文件 / 条目。
这让 Atlas 不只是一个文本文档，而是一个**可导航的知识图**。

## 目录结构

```
Atlas/
├── INDEX.md              ← 唯一入口，所有知识条目在此登记
├── README.md             ← 本文档：Atlas 的思想和使用指南
├── dependency-landscape.md  ← 示例条目：持续维护，topic=build
├── 2026-06-12-some-decision.md  ← 示例条目：ADR，一次性
└── ...
```

目前是扁平结构。如果未来条目超过 20 个，可以按 topic 加子目录；
在那之前，保持扁平——目录层级是**演化出来的**，不是预先设计死的。

## 什么时候往 Atlas 写东西

**值得写进 Atlas 的：**

- 读完了 `docs/agent_guides/` 里某个子系统，写一份「5 分钟速读版」
- 解决了一个问题（升级依赖、修复 build 错误、选了一个方案），把过程 / 结论记下来
- 发现文档和代码不一致，在 Atlas 里标注「以代码为准，这里有偏差」
- 理解了一个模块的组织逻辑，写一份「这个目录为什么长这样」

**不值得写进 Atlas 的：**

- 代码里已经有清晰注释的东西（不要重复造句）
- 完全临时的一次性调试信息
- 可以直接读源码理解的简单事实

## 与项目其他文档的分工

| 文档 | 谁写的 | 稳定性 | 目的 |
|------|--------|--------|------|
| `CLAUDE.md` | 项目维护者 | 静态，长期不变 | AI agent 必须遵守的硬性规则 |
| `docs/` | 项目维护者 | 正式，变更需 review | 正式设计文档，面向所有读者 |
| `docs/agent_guides/` | 项目维护者 | 子系统指南 | 告诉 agent 每个子系统是干什么的 |
| **`Atlas/`** | **任何探索者（人 / agent）** | **持续演进** | **带协作痕迹的速记本，沉淀认知** |

## 如何维护 Atlas

- **每次完成一个非平凡的任务**，回头看 Atlas 中有没有相关的条目需要更新。
  如果有，追加；如果没有，问自己：这件事值得被记住吗？
- **INDEX.md 永远不过时**。新增条目 → 更新 INDEX.md；删除 / 重命名 → 更新 INDEX.md。
  INDEX.md 是 Atlas 健康度的唯一指标。
- **`last-verified` 字段是维护契约**。每个条目的作者对 `last-verified` 负责——
  每次核实后更新日期。reviewer 在 review 时也要检查引用的条目是否已过期。

## 让 Atlas 保持活力的习惯

1. **探索即记录**。每次读一个新子系统，写一段 Atlas 笔记。不用完整，不用正确，
   可以是「目前理解是这样，可能有错」——draft 状态也是贡献。
2. **解决问题后写 ADR**（Architecture Decision Record，中文常叫“架构决策记录”）。选了一个方案而不是另一个，把理由写下来。
   6 个月后你会感谢当时的自己。
3. **把「踩坑记」变成「避坑指南」**。build 失败、依赖冲突、API 变更——
   记录下来，下次同类问题的解决速度会快 10 倍。

---

_Atlas 是一个活的系统。它的价值来自持续的使用和迭代。
第一篇文档不需要完美，只需要存在。_
